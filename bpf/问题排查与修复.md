# eBPF 事件捕获问题排查与修复

本文档记录了在 VMware 虚拟机上 eBPF Tracepoint 调试过程中遇到的问题及解决方案。

---

## 问题概述

VMware 虚拟机上 eBPF 程序加载成功，但用户态无法接收到任何事件。程序运行无报错，但 `ls /tmp` 等命令的执行事件全部丢失。

---

## 问题一：Ringbuf 和 Perf Event Array 混用

### 现象

eBPF 探针加载成功，`link.Tracepoint()` 调用无报错，但 perf reader 始终读不到数据。

### 根本原因

**C 探针使用 Ringbuf 发送事件，但 Go 程序使用 Perf Reader 读取事件。**

这两种机制完全不兼容：
- **Ringbuf**：`bpf_ringbuf_reserve()` / `bpf_ringbuf_submit()` - 内核 5.8+ 专用
- **Perf Event Array**：`bpf_perf_event_output()` + `perf.NewReader()` - 通用机制

### C 探针代码（错误）

```c
// probe_6_0.c - 使用 Ringbuf
struct {
    __uint(type, BPF_MAP_TYPE_RINGBUF);
    __uint(max_entries, 4096);
} rb SEC(".maps");

SEC("tp/sched/sched_process_exec")
int handle_sched_process_exec(...) {
    struct ebpfai_event *e = bpf_ringbuf_reserve(&rb, sizeof(*e), 0);
    // ... fill event
    bpf_ringbuf_submit(e, 0);
}
```

### Go 读取代码（错误）

```go
// main.go - 使用 Perf Reader
rd, err := perf.NewReader(objs.Events, 4096)
// rd.Read() 永远读不到 Ringbuf 的数据
```

### 修复方案

统一使用 Perf Event Array：

**C 探针修改**

```c
// probe_6_0.c - 改用 Perf Event Array
struct {
    __uint(type, BPF_MAP_TYPE_PERF_EVENT_ARRAY);
    __uint(max_entries, 256);
} events SEC(".maps");

SEC("tp/sched/sched_process_exec")
int handle_sched_process_exec(struct trace_event_raw_sched_process_exec *ctx) {
    struct ebpfai_event event = {0};
    event.pid = bpf_get_current_pid_tgid() >> 32;
    event.ppid = ctx->old_pid;
    // ... fill event

    bpf_perf_event_output(ctx, &events, BPF_F_CURRENT_CPU, &event, sizeof(event));
    return 0;
}
```

## 问题二：Perf Reader 不会自动挂载 Tracepoint

### 现象

`perf.NewReader()` 创建成功，但 `link.Tracepoint()` 调用后仍然收不到事件。

### 根本原因

`perf.NewReader()` 只是创建一个读取器对象，**不会自动将 BPF 程序附加到 Tracepoint**。

必须显式调用 `link.Tracepoint()` 来附加 BPF 程序。

### 修复方案

在 `perf.NewReader()` 之前添加 `link.Tracepoint()` 调用：

```go
// 1. 加载 eBPF 对象
var objs bpf.EventObjects
if err := bpf.LoadEventObjects(&objs, nil); err != nil {
    log.Fatalf("failed to load eBPF objects: %v", err)
}
defer objs.Close()

// 2. 显式附加 Tracepoint 程序（关键步骤）
l, err := link.Tracepoint("sched", "sched_process_exec", objs.HandleSchedProcessExec, nil)
if err != nil {
    log.Fatalf("failed to attach tracepoint: %v", err)
}
defer l.Close()
logger.Println("tracepoint attached successfully")

// 3. 现在才能读取事件
rd, err := perf.NewReader(objs.Events, 4096)
```

## 问题三：系统 Tracing 干扰

### 现象

`/proc/sys/kernel/tracing_on` 值为 1，导致 BPF Tracepoint 被阻塞。

### 根本原因

Linux 内核的 ftrace 系统中，`tracing_on` 控制 Tracepoint 的全局开关。当值为 0 时，所有 Tracepoint 被禁用。

### 诊断方法

```bash
# 检查 tracing_on
cat /proc/sys/kernel/tracing_on

# 检查当前事件
cat /proc/sys/kernel/tracing/tracing_on
```

### 修复方案

```bash
# 临时禁用系统 tracing
echo 0 | sudo tee /proc/sys/kernel/tracing_on

# 或使用 setpriv（如果可用）
sudo setpriv --clear-screen -- auxv /proc/sys/kernel/tracing_on 0
```

**注意**：这会临时禁用系统 Tracepoint，可能影响 `perf top` 等工具。测试完成后恢复：

```bash
echo 1 | sudo tee /proc/sys/kernel/tracing_on
```

## 问题四：文件名读取到垃圾数据

### 现象

事件能收到了，但 `filename` 字段包含大量乱码或过期数据。

### 根本原因

`event.Filename` 是固定 256 字节的数组，直接 `binary.Read()` 会读取全部 256 字节，包括未使用的部分。

正确做法是找到 null 终止符，只读取实际字符串。

### 错误代码

```go
// 错误：读取全部 256 字节
var event bpf.Event
r := bytes.NewReader(record.RawSample)
binary.Read(r, binary.LittleEndian, &event.Pid)
binary.Read(r, binary.LittleEndian, &event.Ppid)
r.Read(event.Filename[:])  // 包含垃圾数据
filename := string(event.Filename[:])  // 错误
```

### 修复代码

```go
// 正确：找到 null 终止符
var event bpf.Event
r := bytes.NewReader(record.RawSample)
binary.Read(r, binary.LittleEndian, &event.Pid)
binary.Read(r, binary.LittleEndian, &event.Ppid)
r.Read(event.Filename[:])

// 找到实际字符串长度
filenameLen := 0
for i, b := range event.Filename {
    if b == 0 {
        filenameLen = i
        break
    }
    if i == len(event.Filename)-1 {
        filenameLen = i + 1
    }
}
filename := string(event.Filename[:filenameLen])
```

## 调试检查清单

当事件捕获失败时，按顺序检查：

| 步骤 | 检查项 | 命令 |
|------|--------|------|
| 1 | eBPF 程序是否加载 | `sudo bpftool prog list` |
| 2 | Tracepoint 是否附加 | `sudo bpftool link list` |
| 3 | tracing_on 是否开启 | `cat /proc/sys/kernel/tracing_on` |
| 4 | Perf Event Map 是否有数据 | `sudo bpftool map dump name events` |
| 5 | 内核日志是否有错误 | `sudo dmesg \| grep -i bpf` |
| 6 | 权限是否足够 | `id` (需 root 或 CAP_SYS_ADMIN) |

## 总结

eBPF 事件捕获失败的常见原因：

1. **机制不匹配**：发送端（Ringbuf/Perf Event）和接收端（perf.NewReader/ringbuf.NewReader）必须一致
2. **Tracepoint 未附加**：`perf.NewReader` 不会自动附加程序，需要 `link.Tracepoint()` 显式操作
3. **系统 Tracing 干扰**：`tracing_on=0` 会阻止所有 Tracepoint
4. **字符串解析错误**：固定长度数组需要手动找到 null 终止符
