# eBPF 事件捕获问题排查与修复

本文档记录了在 VMware 虚拟机上 eBPF Tracepoint 调试过程中遇到的问题及解决方案。

---

## 问题概述

VMware 虚拟机上 eBPF 程序加载成功，但用户态无法接收到任何事件。程序运行无报错，但 `ls /tmp` 等命令的执行事件全部丢失。

## 二、本次重构发现的新问题

### 问题 5：BPF Link 占用 tracepoint（无法分离）

**现象**
```
failed to load eBPF objects: field HandleSchedProcessExec: program handle_sched_process_exec: load BTF: BTF not supported
```

**定位过程**
```bash
# 1. 检查 BPF 程序列表
sudo bpftool prog list
# 发现：548: tracepoint  name handle_sched_process_exec

# 2. 检查 BPF Link
sudo bpftool link list
# 发现：10: perf_event  prog 548  tracepoint sched_process_exec
```

**根本原因**
之前测试时的 BPF 程序和 link 仍然附着在 tracepoint 上，阻止了新程序加载。

**解决方案**
内核 6.8 不支持 `bpftool link detach`，必须重启 VM 清理所有 BPF 资源。
```bash
sudo reboot
```

---

### 问题 6：unprivileged_bpf = 2（重启后重置）

**现象**
```
ERROR: unprivileged_bpf=2 (locked), cannot load eBPF programs
```

**定位过程**
```bash
cat /proc/sys/kernel/unprivileged_bpf_disabled
# 输出：2
```

**根本原因**
VM 重启后，`/proc/sys/kernel/unprivileged_bpf_disabled` 重置为 2（锁定）。

**解决方案**
```bash
echo 0 | sudo tee /proc/sys/kernel/unprivileged_bpf_disabled
```

---

### 问题 7：tracing_on = 0（重启后重置）

**现象**
```
WARNING: tracing_on=0, tracepoints are disabled
WARNING: eBPF events will not be captured
```

**定位过程**
```bash
# 检查 tracing_on 位置（Ubuntu 24.04 不同）
cat /sys/kernel/debug/tracing/tracing_on
# 输出：0

# 或者
cat /proc/sys/kernel/tracing_on
# 不存在
```

**根本原因**
1. Ubuntu 24.04 的 tracing_on 在 `/sys/kernel/debug/tracing/tracing_on`
2. VM 重启后重置为 0

**解决方案**
```bash
echo 1 | sudo tee /sys/kernel/debug/tracing/tracing_on
```

### 问题 8：C 源文件导致 Go 编译失败

**现象**
```
imports ebpf-ai-agent/bpf: C source files not allowed when not using cgo or SWIG: probe.c probe_5_8.c probe_6_0.c
```

**根本原因**
bpf2go 生成的 `_bpfel.go` 文件嵌入了 BPF 目标字节码，但项目中的 `.c` 文件仍然存在且 Go 编译器会尝试编译它们。

**解决方案**
为所有 C 文件添加 `//go:build ignore` 构建标签：
```bash
cd bpf/
for f in probe.c probe_5_8.c probe_6_0.c; do
    echo '//go:build ignore' > "${f}.new"
    cat "${f}" >> "${f}.new"
    mv "${f}.new" "${f}"
done
```

### 问题 9：diagnostics.go 读取 tracing_on 路径错误

**现象**
```
WARNING: tracing_on=0, tracepoints are disabled
# 但实际值是 1
```

**根本原因**
Ubuntu 24.04 的 tracing_on 不在 `/proc/sys/kernel/tracing_on`，而是在 `/sys/kernel/debug/tracing/tracing_on`。

**解决方案**
```go
tracingOnPaths := []string{
    "/sys/kernel/debug/tracing/tracing_on",
    "/proc/sys/kernel/tracing_on",
}
for _, path := range tracingOnPaths {
    if data, err := os.ReadFile(path); err == nil {
        state.TracingOn = strings.TrimSpace(string(data)) == "1"
        break
    }
}
```

### 问题 10：Ringbuf map 名称与代码不匹配

**现象**
```
objs.Events undefined (type bpf.EventObjects has no field or method Events)
```

**定位过程**
```bash
grep -A10 "type EventMaps" bpf/event_bpfel.go
# 发现：
type EventMaps struct {
    Rb *ebpf.Map `ebpf:"rb"`
}
```

**根本原因**
bpf2go 生成的代码中，Ringbuf map 名称是 `Rb` 而不是 `Events`。

**解决方案**
```go
// 使用 Rb 而不是 Events
if objs.Rb != nil {
    ringReader, err := ringbuf.NewReader(objs.Rb)
    // ...
}
```

## 三、重构完成的功能

### 1. argv 捕获（结构体扩展）

- `probe.h`: `ebpfai_event` 新增 `argv[512]`
- `bpf/bpf.go`: Go Event 结构体同步扩展

**注意**：argv 实际填充需要 kernel 6.8+ 支持，当前预留字段为空。

- `probe.h`: `filename[256]` → `filename[1024]`
- 实际传输仍按 null 终止符计算长度

### 3. Ringbuf 迁移（kernel 6.0+）

- `probe_6_0.c`: 使用 `BPF_MAP_TYPE_RINGBUF`
- `main.go`: 使用 `ringbuf.NewReader` 读取

- `pkg/diagnostics/diagnostics.go`
- 启动时检查：`kernel version`、`CPU`、`内存`、`tracing_on`、`unprivileged_bpf`、`BTF`
- 监控丢失事件计数

## 四、调试检查清单

当事件捕获失败时，按顺序检查：

| 步骤 | 检查项 | 命令 |
|------|--------|------|
| 1 | VM 是否刚重启（BPF 资源是否清理） | `sudo bpftool link list` |
| 2 | unprivileged_bpf 是否为 0 | `cat /proc/sys/kernel/unprivileged_bpf_disabled` |
| 3 | tracing_on 是否为 1 | `cat /sys/kernel/debug/tracing/tracing_on` |
| 4 | 是否有残留 BPF 程序 | `sudo bpftool prog list \| grep sched` |
| 5 | BTF 是否可用 | `ls /sys/kernel/btf/vmlinux` |
| 6 | 内核日志错误 | `sudo dmesg \| grep -i bpf` |

## 五、验证测试命令

```bash
# 1. 检查系统状态
```

## 六、附录：历史问题

### 问题 A：Ringbuf 和 Perf Event Array 混用

**原因**：C 探针用 Ringbuf，Go 程序用 Perf Reader，机制不匹配。

**修复**：统一使用 Perf Event Array（当时），后来升级为 Ringbuf。

### 问题 B：Perf Reader 不会自动挂载 Tracepoint

**原因**：`perf.NewReader()` 不会自动附加程序。

**修复**：显式调用 `link.Tracepoint()`。

### 问题 C：tracing_on 干扰

**原因**：`tracing_on=0` 禁用所有 Tracepoint。

**修复**：`echo 1 | sudo tee /sys/kernel/debug/tracing/tracing_on`

### 问题 D：文件名乱码

**原因**：固定 256 字节数组未找到 null 终止符。

**修复**：手动扫描 null 终止符。

---

## 总结

eBPF 事件捕获失败的常见原因：

1. **BPF 资源未清理**：残留 link 占用 tracepoint（必须重启）
2. **unprivileged_bpf 锁定**：需 `echo 0 > /proc/sys/kernel/unprivileged_bpf_disabled`
3. **tracing_on=0**：需 `echo 1 > /sys/kernel/debug/tracing/tracing_on`
4. **机制不匹配**：Ringbuf 和 Perf Reader 必须对应
5. **字符串解析错误**：需找到 null 终止符
